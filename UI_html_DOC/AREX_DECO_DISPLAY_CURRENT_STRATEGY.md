# AREX 减压显示当前策略复盘

本文记录这次多气体减压显示问题的排查过程，以及当前分支最终采用的产品层显示策略。重点不是替代算法文档，而是把 UI/适配层为什么这样接算法结果讲清楚，方便后续和算法工程师对齐。

## 1. 问题起点

最初现象是：潜水员配置了多气体，例如 AIR / EAN32 / O2 100%，但实际过程中并不一定按推荐完成每一次切气。日志里会出现当前 active gas 仍是 AIR 或 EAN32，而 `AREX_PLAN` 的站点 gas 已经指向更优气体，例如 O2 100%。这会导致 UI 上的 DECO 时间、进度条和站点看起来像“突然完成”或“突然跳站”。

关键原因是：plan 接口接收的是可用气体计划，算法会基于这些气体寻找最优减压路径；如果某个气体仍被传入且处于 MOD 范围内，算法可以把未来站点规划到这个气体上。算法并不保存“用户刚才拒绝切气”这种产品层策略状态。

## 2. 排查过程

第一步先确认 `gas` 字段含义。日志里 `gas=2` 是算法返回的气体索引，不是 UI 文本。后来增加了 `DECO_GAS_MAP`，把算法 gas index 映射回 UI slot 和气体名，例如 `gas=2(slot2 O2 100%)`，避免只看数字误判。

第二步确认 active gas 是否传入算法。当前适配层会把当前 active gas 映射成 plan/step 的 active index，同时也会把配置中的可用多气体传给算法。也就是说，算法既知道当前正在呼吸哪个气体，也知道后续有哪些气体可用于规划。

第三步确认“忽略气体”策略。产品层如果判断用户错过或拒绝了推荐切气，需要在产品层禁用/移除该气体后重新 plan；core 不负责保存拒绝状态。当前逻辑支持多个气体被忽略，例如 3 gas 中同时忽略两个气体。

第四步排查 `dur / hold / sw`。算法站点返回里：

- `hold_seconds`：真正需要在该深度停留的时间。
- `switch_penalty_seconds`：切气惩罚/切气耗时。
- `duration_seconds`：`hold_seconds + switch_penalty_seconds`。

因此当日志出现 `dur=60s hold=0s sw=60s` 时，它不是一个真实减压停留，而是一个纯切气预测站。如果 UI 直接把 `duration_seconds` 当 DECO 倒计时显示，就会把切气动作时间误显示成减压停留时间。

## 3. 曾尝试和舍弃的方向

曾考虑把 `gas_switch_penalty_seconds` 改成 0，这样 `duration_seconds` 不再包含切气惩罚，UI 上也不会看到纯切气 60 秒。但这个做法会改变传给算法的规划条件，相当于让算法认为切气没有时间成本。

最终没有采用这个方向。当前分支保留算法配置里的切气惩罚，让算法仍按真实产品策略规划；显示层再选择只展示真实停留时间。

## 4. 当前最终策略

当前分支采用：

- `DECO_GAS_SWITCH_PENALTY_SECONDS = 60U`：算法 plan 仍保留 60 秒切气惩罚。
- `DECO_HIDE_SWITCH_ONLY_STOPS = (DECO_GAS_SWITCH_PENALTY_SECONDS > 0U)`：开启后隐藏纯切气预测站。
- UI/NDL 样式 DECO 时间使用 `hold_seconds`，不再使用 `duration_seconds`。
- `hold=0 sw=60` 的纯切气站不进入 UI 可画站点。
- 日志仍保留 `dur / hold / sw` 全字段，便于继续判断算法原始输出。

这套策略的核心是：算法可以继续按“完整可用气体 + 切气成本”规划，但 UI 只把真实停留时间显示给用户，避免把切气惩罚伪装成减压倒计时。

## 5. NDL/停留时间显示策略

当前新增宏：

```c
#define UI_NDL_STOP_TIME_MINUTE_ONLY  1U
```

开启后，`COMP_NDL_STOP_1606` 和 `COMP_STOP_TIME_1606` 的停留倒计时只显示分钟，并且按秒数向上取整。例如：

- `1s` 到 `60s` 显示 `1`
- `61s` 到 `120s` 显示 `2`
- `0s` 显示 `0`

这样做是为了避免秒级倒计时和算法重规划之间出现视觉误导。减压计划本身每 tick 可能重算，秒数抖动很容易让用户误以为“还有几十秒却突然结束”。分钟向上取整更接近潜水电脑常见显示习惯，也更稳定。

## 6. 当前仍需要和算法侧确认的问题

1. 当用户没有切到某个更优气体，但该气体尚未被产品层标记 ignored 时，算法继续把该气体纳入 plan 是否符合预期。
2. 产品层判断“错过推荐气体”的精确定义：是超过 MOD/推荐深度 1m 后立即 ignored，还是需要结合状态机 INFO 消失事件。
3. `gas_switch_penalty_seconds` 是否应该始终由产品配置决定，还是不同模式下由算法库默认值决定。
4. 如果 active gas 与未来最优 gas 不一致，算法返回站点 gas 是否只代表“推荐使用气体”，而不代表当前实际呼吸气体。

当前 UI 策略不会掩盖这些问题：日志仍会打印算法原始 gas、active gas、`dur/hold/sw` 和 gas map。显示层只是避免把非停留时间当作 DECO 时间画出来。
