# 规则书

本文档是本仓库固定规则书。以后说“写规则里面”或“写到规则里”，默认就是更新本文档；如果某条规则已经有专题文档，也要同步更新对应专题文档。

## 文档同步规则

- 任何影响算法计算口径、算法字段语义、算法输入输出映射、模拟器与真机计算同步的改动，都必须同步写入 Markdown 文档。
- 优先更新本文档；如果属于 AREX 算法适配细节，同时更新 `UI_html_DOC/AREX_ALGO_SIM_SYNC_LOG.md`。
- 如果改动影响 API 调用链、推荐气体、停站/计划、组织仓/GF、MOD、nofly、RTC 离线补偿等专题，也要同步更新对应专题文档。
- 文档必须说明：改了什么、为什么改、旧口径和新口径差异、模拟器和真机侧各自需要注意什么、验证方式。

## 回调移植规则

- `src/ui/core/callbacks.c` 中的弱回调不是“只给 PC 模拟器用”的临时逻辑；它们是 UI 菜单/设置进入业务层的接口样例。
- 真机侧可以覆盖弱回调，或用等价平台实现接管，但必须保持相同业务语义。
- 新增或修改设置项时，如果该设置会影响算法输入、派生计算、持久化配置、传感器/平台状态或告警策略，必须检查对应 `ui_on_*` 回调。
- 不能只改 `submenu_model.c` 的显示状态；最终必须落到明确的业务回调、`bus_set_*()`、平台接口或算法适配接口。
- 如果 PC 默认回调和真机强实现需要不同处理，必须在规则书或移植文档中写清楚差异。
- `Time/date / 24-hour` 会影响全局当前时间显示，菜单动作必须调用 `ui_on_time_24h_set(bool enabled)`，默认实现写入 `bus_set_time_24h_enabled()` 并触发 `DIRTY_SYSTEM`；真机侧若覆盖回调，需要同步持久化配置，并保证 `COMP_TIME_1606` / `COMP_SYS_1606` 在 12 小时制显示 AM/PM。
- `Time/date / TIME / HOUR-MINUTE` 必须和当前时间组件使用同一个系统时间源。确认 HOUR 或 MINUTE 时必须通过 `ui_on_datetime_field_set()` 写入 `bus_set_sys_time()`；菜单摘要再次打开时也应从 `bus_get_sys_time_h/m()` 回填，不能只维护菜单本地缓存。
- `DISPLAY / TEMP` 只切换温度显示单位，传感器和 data bus 原始温度仍按摄氏度保存。菜单动作必须调用 `ui_on_temperature_unit_set(uint8_t unit)`，默认实现写入 `bus_set_temperature_unit()` 并触发 `DIRTY_SYSTEM`；真机侧若覆盖回调，需要同步持久化配置，并保证主界面、SYS、INFO 和自定义温度组件在 C/F 间一致刷新。

## 告警交互规则

- 告警条件不属于 `bus_set_*()`；算法、平台任务或调试层负责调用 `alarm_set_active()` 设置或解除条件。
- 告警确认统一走 `alarm_confirm_current()`，PC TCP 使用 `back 2`，真机侧应映射到长按 BACK 两秒。
- `CRIT` 告警确认只隐藏横幅，不取消原位模块闪烁；原位模块必须等条件解除后恢复。
- `WARN` 告警确认只隐藏横幅，不取消原位模块呼吸；原位模块必须等条件解除后恢复。
- 普通 ENTER 不确认告警，也不执行推荐气体切换；普通 BACK 只做 UI 返回。
- `INFO_GAS_SWITCH` 使用 `back 2` 确认，确认后自动提交推荐气体切换请求。
- 普通 INFO 横幅显示 5s 后自动消失；`INFO_GAS_SWITCH` 不走 5s 自动消失。
- GAS SWITCH 提示的上浮消失阈值由 `ALARM_GAS_SWITCH_PROMPT_EXIT_DELTA_M` 控制，默认 `1.0f`。

## 算法参数同步规则

- 会影响算法计算的参数，修改后必须同步到算法适配层，并触发必要的输出刷新。
- 已知会影响算法或派生显示的参数包括：GF/保守度、盐度、水型、last deco stop、safety stop、气体模式、O2/He、每槽最大 PO2、MOD、nofly、组织仓/GF 指标、RTC 离线补偿。
- 参数变更后如果只是更新 `g_sys_config` 或菜单缓存，而没有更新算法 config/gas plan/data bus，就是不完整修改。
- 如果参数会影响缓存值，例如 gas slot MOD、TTS、停站列表、推荐气体、组织仓/GF、nofly，必须确认配置变更后这些缓存会刷新。
- 已有算法接口时，主工程必须直接调用算法接口，不允许在 UI 或模拟器里复刻公式。
- 如果算法层没有提供接口，不能随手在 UI 侧补公式；必须先记录边界，等待算法或平台给明确输入/输出契约。

## 潜水生命周期与水面 AIR 规则

- 入水、潜水中、疑似出水、已确认水面必须由平台/模拟器生命周期状态机判断，算法适配层只消费“已确认水面”状态，不用深度阈值自己猜状态。
- PC 模拟器当前入水确认规则：`depth >= 0.5m` 连续 `3` 个模拟秒后进入潜水；入水确认中仍按水面 AIR 计算，确认进入潜水后才使用真实深度和当前 active gas。
- PC 模拟器当前出水确认规则：`depth <= 0.2m` 连续 `30` 个模拟秒后进入已确认水面。
- TCP `speed` 是模拟时间倍速，必须同时加速入水/出水确认计时；例如 `speed 30` 下，30 个模拟秒约等于 1 个真实秒。
- 已确认水面后，实时 `deco_core_tick()`、`arex_deco_plan()` 和 `rtc_offline` 都必须按 `0m + AIR` 计算组织恢复；不能继续使用 UI 当前选择的多气体 active gas。
- 已确认水面后，组织图输出也必须按同一份 `0m + AIR` 临时状态调用 tissue pressure metrics；`tissue_pi_permille` 不能跟随 UI 当前 active gas 或气体编辑变化。
- 水面 AIR 只影响算法恢复计算，不改写 UI 的当前气体选择、气体槽配置或 active gas 显示。
- NOFLY 显示必须有产品生命周期门控：上电、刚连接 TCP、`SURFACE_CONFIRMED` / `ENTRY_PENDING` 且尚未确认进入过潜水时显示 `00:00`，不能把算法 no-deco 最小禁飞策略的 `12h` 显示成未潜水禁飞义务；确认进入过潜水后才消费 `arex_deco_nofly()` 返回值。
- `rtc_offline` 只允许在已确认水面执行，输入算法的深度固定为 `0m`，气体固定为 AIR。
- PC TCP 调试水面离线恢复只能使用 `rtc_offline <seconds>`；旧 `surface <seconds>` 只改 UI 计时、不推进算法，已删除。
- 真机侧移植时需要用自己的传感器和潜水生命周期判断调用 `deco_core_set_surface_confirmed()` 或等价接口；不能只照搬 PC 模拟器阈值，也不能把“出水判断”散落在算法调用点。

## 停站与告警字段规则

- `ArexDecoSchedule.ceiling_violated` 是算法层明确输出的违规标志，产品层必须消费；不能只通过 `stops == 0`、`tts == 0` 或 UI 自己的 depth/ceiling 比较来判断是否安全。
- Runtime 主界面的当前停站倒计时使用 `ArexDecoStop.duration_seconds`，保持和 planner/TTS 同一口径。
- `hold_seconds` 是物理停留时间，`switch_penalty_seconds` 是 planner 预测切气延迟；二者可以用于调试或详情拆分，但不要替代主倒计时。
- DIVE PLAN 结果页的后端算法只返回 `schedule.stops[]` 和 `tts_seconds`，不会返回独立上升段列表；UI/适配层只显示“计划底部深度到第一个可见停站”的 `asc` 展示行，不显示后续停站之间或最终升水的 `asc` 行，也不得把展示行回写为算法停站。
- DIVE PLAN 不显示最终 `0m asc` 行；最后一段升水时间和气量仍计入 Runtime / Gas 汇总，避免列表末尾出现非停站的 0m 假行。
- 当某个 stop 满足 `hold_seconds == 0 && switch_penalty_seconds > 0` 时，它是纯切气预测站，不能渲染成右上角 `DECO xxm 0:00` 或实时轨迹减压站；切气提示由推荐气体链路和产品层门控处理。
- `DECO_HIDE_SWITCH_ONLY_STOPS` 只控制纯切气预测站是否进入 UI 停站展示；不得用它改写算法 schedule、TTS、氧暴露、气量估算或真实 active gas 状态。
- 安全停留 runtime 显示归算法状态接口所有：右上角 `SAFE` 必须读取 `arex_deco_safety_stop()` 的 `required/counting/remaining_seconds/phase`，不得再从 `arex_deco_plan().stops[]`、`max_depth/current_depth/ndl` 或 UI 本地安全区间自行推断。`arex_deco_plan()` 中的 safety stop 只用于 TTS/计划/气量预测路径。
- `arex_deco_plan()` 返回非 OK 时，不能把它等价为 `stops=0` 或“无安全停留/无减压站”。如果 `arex_deco_step()` 已成功，本轮只同步非 schedule 数据；TTS、右上角当前停站、实时轨迹停站列表和 `ceiling_violated` 告警保持上一帧有效 plan 输出。
- 气体密度、TTS hold 预测和 NDL 上下 3m 预测必须优先调用算法接口：`arex_deco_calculate_gas_density()`、`arex_deco_forecast_tts_hold()`、`arex_deco_forecast_ndl_excursion()`。其中 TTS hold 属于高开销预测，固件侧按低频刷新；TTS 预测失败时保留上一帧显示值，初始化默认显示 0。算法返回的 `tts_delta_hold_seconds` 保持秒级严格相减；UI 的分钟级 `TTS Δ +5min` 为保证视觉可加，显示为 `round_display(TTS @ +5min) - round_display(current TTS)`。
- `NDL↑3` / `NDL↓3` 来自算法上下 3m 预测；`NDL△3` 是固件侧动态显示聚合项，不是算法输出指标，也不是 `预测 NDL - 当前 NDL`。固件根据潜水员当前升降状态选择显示 `NDL↑3` 或 `NDL↓3`，方向不明确或预测失败时显示 `--`，不得保留上一帧伪装成当前方向。
- DECO 卡片 16 组织仓主图使用归一化 data bus 字段：`tissue_bar_permille[16]` 作为 0~1000 条长，400 是环境压力线，900 是 M 值线，`tissue_pi_permille` 是吸入总惰性气体分压虚线。`tissue_bar_permille[16]` 按总惰性气体压力 `PN2 + PHe` 计算，不允许 UI 只用 `PN2_i` 自行重算条长；完整计算说明见 `UI_html_DOC/16组织仓归一化视图计算说明.md`。
- AREX Deco Core `0.0.23` 起，组织压力真值必须通过 `arex_deco_calculate_tissue_pressures()` 获取；适配层不得再用 `absolute_gf_percent[16]` 反推 M 值。`tissue_m_value_bar[16]`、`tissue_m_gf_bar[16]`、`tissue_he_bar[16]`、`tissue_inspired_he_bar` 都属于算法 pressure metrics 的消费结果。
- `COMP_TISSUE_RAW_4012` / `COMP_TISSUE_GF_4012` 自定义小组件也显示横向 16 行组织仓，不再提取单条 Leading Tissue。RAW 小组件直接消费 `tissue_bar_permille[16]`，400 是环境压力线，900 是 M 值线；GF 小组件使用 `tissue_gf_pct[16]` 映射条长，400 是环境压力线，900 是当前 target GF 红线。两个小组件都画 400、900 和 `tissue_pi_permille` 参考线，但不显示底部 `PI/PAMB/M` 字母标签。
- `tissue_raw_pct[16]` / `tissue_gf_pct[16]` 保留给自定义小组件和信息概览兼容，不驱动 DECO 主图；其中 `tissue_gf_pct[16]` 是 GF 模式小组件条长映射的直接输入。
- 上述字段语义变更必须同步到 `UI_html_DOC/AREX_ALGO_SIM_SYNC_LOG.md` 和 `UI_html_DOC/AREX_DECO_API_USAGE.md`。

## 气体与 PO2 规则

- 实时 PPO2 和气体最大 PO2 是两个不同概念，不能混用。
- 实时 PPO2 来自当前深度/压力下的运行态输出，用于 PPO2 组件或告警等实时显示；`bus_set_ppo2()` 的单位固定为物理 `bar`。
- 若上游只有压力传感器 mbar，必须用 `bus_calculate_ppo2_bar(o2_pct, pressure_mbar)` 或等价公式 `FO2 * pressure_mbar / 1000.0f` 写入实时 PPO2；不得用 `pressure_mbar / 1013.25f` 的 ATA 归一值写入 `ppo2[]`。
- AIR 在 1013.25 mbar 岸上应显示约 `0.21 bar`；10 m 默认海水口径约为 `0.42 bar`。如果产品未来要显示 ATA，必须另起字段或明确单位，不得复用实时 PPO2 bus。
- 气体最大 PO2 是 gas slot 配置值，用于 MOD、算法 gas plan 的 `max_ppo2_bar`、切气/计划口径。
- 多气体模式中，每个 gas slot 可以有独立最大 PO2；算法适配层必须优先使用每槽 PO2。
- 全局 `DIVE SETUP / MOD PO2` 只能作为旧路径或未配置槽 fallback，不能覆盖已配置气体槽自己的 PO2。
- `3 GAS` 和 `OC Tech` 的每个可编辑气体槽有独立 `ACTIVE: ON/OFF` 状态。`ACTIVE: OFF` 表示该槽从运行 gas profile 中移除，不进入算法 gas plan、推荐气体、切气列表或 DIVE PLAN 计算；但 O2/He/PO2 配置值仍保留，重新切回 ON 后继续使用原配置。
- `AIR` 和 `NITROX` 是单气体模式，不显示 `ACTIVE`，也不允许通过该入口删除唯一气体。
- 真机侧如果覆盖设置回调，必须同步实现 `ui_on_three_gas_active_set(uint8_t slot, bool active)` 和 `ui_on_oc_tech_active_set(uint8_t slot, bool active)`，并把 active 状态纳入持久化配置快照；否则 UI 菜单状态、算法 gas plan 和重启后的配置会不一致。

## 真机算法计算回调规则

- `bus_calculate_gas_mod()` 是 UI/data 层请求 MOD 计算的统一入口。
- PC 模拟器路径可以直接调用 `deco_core_calculate_gas_mod()`；真机非 PC 路径必须通过 `ui_calculate_gas_mod()` 回调接到真实算法实现。
- 真机侧移植时必须实现或覆盖 `ui_calculate_gas_mod(uint8_t o2_pct, uint8_t he_pct, float max_ppo2)`，不能让默认弱实现长期返回 `0.0f`。
- 如果真机算法库无法计算 MOD，必须在移植文档中记录 fallback 口径和风险，不能在 UI 层私自复刻公式。
- 算法静态库和头文件必须做版本一致性检查。PC 适配层使用 `arex_deco_get_api_version()` 对比 `AREX_DECO_API_VERSION_*`，真机侧也应保留等价检查，避免头文件和 `.a` 版本不一致时继续运行。

## 当前待明确规则

- `ALTITUDE` 当前只保存 UI 配置，AREX 适配层尚无“altitude level -> surface_pressure_bar”的明确算法/平台接口。按本规则，主工程暂不复刻气压换算公式；后续需要算法或平台层提供明确映射后再接入。
