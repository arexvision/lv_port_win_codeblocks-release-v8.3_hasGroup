# 潜水算法真机与模拟器数据链路对照

日期：2026-06-18

范围：仅检查当前仓库 `src` 下 SF32 嵌入式端与 PC 模拟器中的潜水算法相关链路；不检查 `diver-app/`、驱动层、外部 SDK；本次只做代码对比和文档记录，不修改业务代码。

## 结论摘要

当前真机链路和模拟器算法链路使用的是同一套 AREX core API，核心变量大部分能对应上，但还不能说“所有数据拿取方式完全相同”。主要差异集中在 5 类：

1. 真机用压力 `mbar` 调 `arex_deco_step_pressure()`，模拟器用深度 `m` 调 `arex_deco_step()`。
2. 真机减压计划有缓存/节流刷新，模拟器每个 tick 都重新 `arex_deco_plan()`。
3. 安全停留 `CNTUP` 模式真机是独立正计时逻辑，模拟器仍按 180 秒倒计时配置给 core。
4. PO2 有两种口径并存：真机 active PO2 和模拟器 per-slot PO2 用 `pressure_bar = mbar / 1000`，真机气体槽/日志/设置侧局部代码使用 `pressure_mbar / 1013.25`。
5. 真机潜水生命周期有入水/出水防抖，模拟器生命周期更直接，轨迹和潜水时间轴天然会有差异。

因此，从代码审计角度看：核心算法已接入，但仍建议在实际下水前至少完成同一 profile 的真机压力模拟、PC 模拟器输出对账、浅水验证。仅凭本次静态对比，不建议把它当作最终下水放行结论。

## 真机链路

主链路：

```text
src/app/tasks/algorithm/arex_deco_task.cpp
  -> read_current_depth_frame()
  -> update_auto_dive_state()
  -> decompression_control_apply_pending()
  -> ArexDecoRuntime::progressDive(pressure_mbar, delta_s)
  -> arex_deco_bridge_publish()
  -> build_payload()
  -> Core::DataBus::SetDecompression()
  -> ui_adapter_data_bridge.cpp::ApplyDecompressionFields()
  -> app_ui bus_set_* / bus_update_deco()
```

真机关键点：

- 深度/压力来自 `DepthService` 快照或串口 debug 注入，算法步进最终使用绝对压力 `pressure_mbar`。
- 自动潜水状态：大于约 `1.2m` 持续约 `2s` 才进入潜水，小于约 `0.8m` 持续约 `30s` 才出水。
- `elapsed_algorithm_ms` 由算法任务 tick 计算，debug 注入时会乘 `time_scale`。
- `delta_time_seconds` 用整数秒推进 AREX core，最小保证为 `1s`。
- UI 发布经 `DecompressionPayload`，再进入 `bus_set_*`。
- 当前代码中实时 UI 轨迹由 `arex_deco_log_adapter.cpp::append_ui_track_if_needed()` 按 `payload.is_diving + depth_valid + elapsed_algorithm_ms` 写入，不再依赖 SD 日志 writer 是否已开始。

## 模拟器链路

算法模拟链路：

```text
src/app_ui/hal_sim/sim_data.c
  -> sim_lifecycle_tick(depth_m)
  -> deco_core_tick(depth_m, temperature_c, 1)
  -> src/app_ui/algo_sim/deco_core.cpp
  -> arex_deco_step()
  -> arex_deco_plan()
  -> bus_set_* / bus_update_deco()
```

模拟器需要分清两条路径：

- `src/app_ui/algo_sim/deco_core.cpp` 是真正可对比的算法模拟路径。
- `src/app_ui/hal_sim/sim_data.c` 中部分 demo 数据会直接写 `bus_set_tts()`、`bus_set_deco_plan()`、`bus_set_ppo2()` 等，这些是 UI 演示数据，不应作为 AREX 真机算法对账依据。

| 日志 PO2 | 真机日志高 PO2 判断也使用 `pressure_mbar / 1013.25` | 模拟器告警读取 bus PO2 | 不一致风险 | 与 active PO2 口径不完全一致。 |

| 气体密度 | 真机 `calculateActiveGasDensity(depth,temp)` | 模拟器 `arex_deco_calculate_gas_density()` | 对齐 | 前提是温度、深度、气体一致。 |
| 轨迹图 | 真机按算法 payload 的潜水态和 elapsed 写 UI track | 模拟器 `sim_lifecycle_tick()` 每秒 `dive_log_append_sampled()` | 已接近但不完全一致 | 真机起点受 2s 入水确认影响，模拟器从入水阈值立即记点。 |
| SD 持久日志 | 真机依赖 writer、next dive id、采样间隔 | 模拟器多为 UI 内存轨迹 | 不应直接对比 | UI 轨迹和 SD 日志是两条链路。 |
| 告警 | 真机来自 payload/log/runtime 多处 | 模拟器有 `sim_alert_policy` 直接读 bus | 部分对齐 | PO2 口径差异会影响告警边界。 |

## 当前已看到的关键修正点

这些是当前源码里已经能看到的行为，不是本次文档新增修改：

- `ArexDecoRuntime::syncConfigToState()` 保留实时 `state_.config = config_`，安全停留配置不会在 live state 被清掉。
- `ArexDecoRuntime::rebuildPlan()` 使用 `planner_state = state_` 的临时副本，并只在副本里把 `safety_stop_seconds/enabled` 清零。这样计划列表不把安全停留当强制减压站，但实时安全停留状态仍可由 core 判断。
- `arex_deco_log_adapter.cpp::append_ui_track_if_needed()` 让实时 UI 轨迹只依赖算法潜水态，不依赖 SD 日志 writer 启动成功与否。

## 需要重点验证或后续统一的差异

### 1. PO2 口径不统一

当前存在两类计算：

```text
物理 bar 口径：O2_fraction * (pressure_mbar / 1000.0)
ATA 归一口径：O2_fraction * (pressure_mbar / 1013.25)
```

真机 active PO2 与模拟器 per-slot PO2 使用物理 bar 口径；真机气体槽 PO2、日志高 PO2 判断、部分设置表面 PO2 使用 ATA 归一口径。若 UI 显示、告警、日志希望和模拟器完全一致，后续需要统一口径。

### 2. CNTUP 安全停留模式不一致

真机 `CNTUP` 是正计时，进入安全停留区域后累计 elapsed；模拟器 `UI_SAFETY_STOP_CNTUP` 当前仍作为 `180s` 倒计时配置交给 core。这个会导致同样设置下，安全停留样式、剩余时间、总时间含义不一致。

### 3. 计划刷新频率不同

真机为了降低 `arex_deco_plan()` 的实时负载，使用：

- deco active 时较短周期；
- 普通潜水时较长周期；
- 水面更长周期；
- 深度变化、deco 状态变化、NDL bucket 变化时提前刷新。

模拟器是每 tick 直接 `arex_deco_plan()`。所以 TTS、deco plan、ceiling violated、推荐气体等依赖 schedule 的字段，在真机上可能短时间滞后。这是工程上的性能取舍，但不是完全同取数方式。

### 4. 潜水生命周期不同

真机有入水和出水确认，模拟器更像 UI 直接演示。直接后果：

- 真机轨迹前几秒可能没有开始记点；
- 真机 dive time 与模拟器差 2 秒以上是合理的；
- 出水后真机仍会在防抖窗口保持潜水态更久；
- 推荐气体、安全停留、日志事件的出现时机可能不同。

### 5. 高度/表面压力需要单独对账

真机有 `setAltitude()`，会影响 `surface_pressure_bar`。模拟器算法链路当前主要从 UI salinity/GF/gas/safety 取配置，未看到同等高度压力配置完整链路。若用户设置高度档位，真机和模拟器必须做专门对账。

### 6. `sim_data.c` demo 写数不能作为算法真值

`sim_data.c` 里存在直接写 UI bus 的演示字段，包括 PO2、TTS、deco plan、CNS、OTU、组织图等。这些只适合 UI 视觉验证，不适合判断真机算法是否一致。对账必须固定使用 `deco_core_tick()` 这条 AREX 算法路径。

## 建议验收矩阵

同一 profile 输入下，建议同时记录 PC 模拟器和真机 payload/bus 输出：

| 场景 | 输入 | 需要对账字段 | 允许误差建议 |
| --- | --- | --- | --- |
| 水面 AIR | 0m，AIR 21/0，默认 GF | active gas、slot PO2、MOD、NDL、stop kind | PO2 需先决定口径；其余应一致 |
| 10m 恒深 5min | AIR，盐水，GF 固定 | depth、pressure、NDL、CNS、OTU、GF99、surfGF、track 点数 | NDL/TTS 1min，PO2 0.01bar |
| 20m 恒深 20min | AIR 或 NX32 | NDL、TTS、ceiling、deco plan、safety stop | plan 可有刷新滞后，但稳定后应一致 |
| 安全停留 3min | 下潜超过触发深度后回到 3-6m | stop kind、stop depth、left/total、in zone | 时间 1-2s |
| 安全停留 CNTUP | 同上，mode=CNTUP | 正计时/倒计时行为 | 当前预期不一致 |
| 出水 | 深度回到 <0.8m | dive state、track reset、surface time、log end | 真机出水确认约 30s，不应按模拟器 5s 对齐 |
| 切气 | 多气体，深度进入推荐区 | active gas、recommended gas、MOD、PO2、gas density | index 必须按压缩后槽位对齐 |
| 高度档 | 非 0 altitude | surface pressure、depth-pressure、PO2、NDL/TTS | 必须单独定义表面压力基准 |

## 建议下一步

1. 先固定一个“算法对账 profile”：深度序列、温度、GF、盐度、高度、气体槽、active gas、safety mode。
2. 在 PC 模拟器中只走 `deco_core_tick()`，导出每秒 bus 字段。
3. 在真机算法任务里用同一 profile 注入压力/深度，导出 `DecompressionPayload`。
4. 按字段表比较：先比 core 指标，再比 UI 发布字段，最后比轨迹和日志。
5. 对 PO2、CNTUP、高度、计划刷新频率这 4 类差异先定产品口径，再决定是否修改代码。

## 下水前建议

从静态代码检查看，核心算法链路已经不是“完全断开”或“模拟器假数据直接搬到真机”的状态；但仍有明确口径差异。建议在真实下水前至少完成：

- 同 profile 的 PC/真机逐秒对账；
- 压力桶或可控压力模拟测试；
- 安全停留、NDL 进入/退出、出水确认、轨迹刷新专项验证；
- 浅水短潜验证，且保留独立备份潜水电脑。

在 PO2 口径、CNTUP、安全/计划刷新、轨迹时间轴这些点确认前，不建议把当前静态审计结论视为最终下水放行。
